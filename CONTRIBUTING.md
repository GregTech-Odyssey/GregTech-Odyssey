# Contributing Guidelines / 贡献指南

感谢你愿意为 GregTech Odyssey 做贡献。

本文件适用于向**本项目下任何仓库**提交 Pull Request 的贡献者。  

如果你只是提出问题或建议，请前往 [Issues](https://github.com/GregTech-Odyssey/GregTech-Odyssey/issues)。

> 本指南参考了对应仓库的 AI 使用政策： https://github.com/ghostty-org/ghostty/blob/main/AI_POLICY.md 和 https://github.com/GTNewHorizons/GTNH-Dev-Doc/blob/master/AI_POLICY.md
>
> 本指南由 AI 改写，人类维护者已审核。

## 基本原则

- 贡献者必须理解自己提交的内容，为自己提交的内容负责。
- Bugfix 优先使用最小改动。
- 新功能、平衡性改动，请先确认项目中没有已有实现或现成工具。
- 大型改动，建议先开 Issue / Discussion 说明方案。

## 基本 PR 要求

提交 PR 时，如果涉及到以下部分，请尽量说明：

- 你改了什么
- 为什么需要这个改动
- 如何测试或验证
- 是否影响存档兼容、平衡、性能、任务线或客户端/服务端行为

## AI 使用规则

### 1. 贡献者必须理解自己的改动

你需要为你的改动负责，即使它是由 AI 生成的。

如果你无法在不依赖 AI 工具的情况下解释你的改动做了什么、为什么这样做、以及它会如何影响项目中的其他系统，请不要提交这部分改动。

这适用于代码、配置、文档、翻译和其他项目内容。

### 2. 不要提交自己没有理解或验证的 Bugfix

如果你没有亲自遇到、复现过某个 Bug，也不熟悉相关代码，请不要把问题丢给 AI 后直接转交一个补丁。

这类补丁很容易修错地方，或者制造新的问题。

提交 Bugfix 时，请至少说明：

- 问题如何复现
- 你认为根因是什么
- 你改了什么
- 你如何验证修复有效

### 3. 文档、任务和文本内容必须保留你的意图和语气

AI 可以帮助润色、整理或改写句子，但文本的观点、目的和语气必须来自贡献者本人。

对于任务，**不接受 AI 生成文本**。

### 4. Issue 和讨论可以使用 AI，但必须经过人工处理

你可以使用 AI 辅助整理 Issue、讨论内容或说明文字，但提交前必须由人类认真检查、编辑和压缩。

不要提交未经筛选的 AI 长篇输出。贡献者需要自己做调查，去掉噪音，只保留和问题相关的信息。

### 5. 对 AI 生成媒体资源的保守态度

若您需要提交 AI 生成的媒体资源，包括但不限于：

- 贴图
- 图片
- 视频
- 音频
- 图标或其他美术资源

请确保其质量，且最好先开 Issue / Discussion 说明方案。我们会谨慎评估这些资源是否适合项目使用。

### 6. 滥用 AI 可能导致贡献被拒绝

明显由 AI 生成、缺乏理解、缺乏验证、质量低下的贡献可能会被要求重写、关闭，或在后续贡献中被更严格审查。

### 7. 这里是人在维护的项目

请记住，本项目由真实的人维护。

每一个 Issue、Discussion 和 Pull Request 都需要维护者阅读、判断和审查。提交低努力、未验证、无法解释的内容，本质上是在把验证成本转嫁给维护者。

我们愿意帮助新人学习和成长。

如果你想学习，请直接提出问题、展示你的尝试和理解，而不是用 AI 生成一份自己也不理解的改动。
